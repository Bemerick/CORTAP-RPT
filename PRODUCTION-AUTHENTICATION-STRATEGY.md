# Production Authentication Strategy

## The Problem

**Question:** Is creating a service account going to require saving and updating a password in the Lambda function? How do we handle this for production?

**Short Answer:** NO - we should NOT use service account credentials in Lambda. There's a much better approach!

---

## Why Service Account in Lambda is BAD ❌

### Security Issues

1. **Credentials in Environment Variables**
   - Environment variables are visible in AWS Console
   - Anyone with Lambda read access can see them
   - Hard to rotate without redeploying Lambda

2. **Credential Rotation**
   - Password changes require Lambda redeployment
   - Or use AWS Secrets Manager (adds complexity)
   - Still requires Lambda to authenticate itself

3. **Audit Trail**
   - All reports show as "generated by cortap_service"
   - Can't track which Riskuity user requested the report
   - Compliance and audit issues

4. **Scope Creep**
   - Service account might need broad permissions
   - One compromised token = access to all projects
   - No fine-grained access control

---

## The BETTER Approach: Token Pass-Through (What We Built) ✅

### Architecture

```
┌──────────────────┐
│ Riskuity User    │  1. Logged in, has session
│ (Browser)        │
└────────┬─────────┘
         │ 2. User action: "Generate Report"
         │
         ▼
┌──────────────────┐
│ Riskuity Backend │  3. Generate API token FOR THIS USER
│ (Node/Python)    │     (using service account OR user creds)
│                  │
└────────┬─────────┘
         │ 4. Pass user-specific API token
         │    Authorization: Bearer <USER_API_TOKEN>
         │
         ▼
┌──────────────────┐
│ CORTAP Lambda    │  5. Use provided token (no credentials needed!)
│ (Our Service)    │     Just pass it through to Riskuity API
└────────┬─────────┘
         │ 6. Call Riskuity with user's token
         │
         ▼
┌──────────────────┐
│ Riskuity API     │  7. Token identifies specific user
│                  │     Audit trail shows actual user
└──────────────────┘
```

**Key Insight:**
- ✅ Lambda never stores credentials
- ✅ Lambda just passes through the token it receives
- ✅ Riskuity backend handles authentication
- ✅ Proper audit trail (knows which user)

## Implementation Options

### Option 1: Backend Token Generation (RECOMMENDED) ✅

**Riskuity implements a backend endpoint that generates tokens**

#### Riskuity Backend (Node.js example)

```javascript
// Riskuity Backend - New Endpoint
// File: /api/cortap/generate-report

app.post('/api/cortap/generate-report', async (req, res) => {
  const { projectId } = req.body;
  const user = req.user; // From session middleware

  try {
    // Step 1: Generate API token for this user
    // Option 1A: Use service account (simple)
    const apiToken = await generateServiceAccountToken();

    // OR Option 1B: Generate user-specific token (better audit)
    // const apiToken = await generateUserAPIToken(user);

    // Step 2: Call CORTAP Lambda
    const response = await fetch(
      process.env.CORTAP_LAMBDA_URL + '/api/v1/generate-report-sync',
      {
        method: 'POST',
        headers: {
          'Authorization': `Bearer ${apiToken}`,
          'Content-Type': 'application/json'
        },
        body: JSON.stringify({
          project_id: projectId,
          report_type: 'draft_audit_report'
        })
      }
    );

    const report = await response.json();
    res.json(report);

  } catch (error) {
    res.status(500).json({ error: error.message });
  }
});

// Service account token (cached)
let cachedServiceToken = null;
let tokenExpiry = null;

async function generateServiceAccountToken() {
  // Check if cached token is still valid
  if (cachedServiceToken && tokenExpiry > Date.now()) {
    return cachedServiceToken;
  }

  // Get new token
  const response = await fetch('https://api.riskuity.com/users/get-token', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({
      username: process.env.RISKUITY_SERVICE_USERNAME,
      password: process.env.RISKUITY_SERVICE_PASSWORD
    })
  });

  const data = await response.json();

  // Cache for 55 minutes (tokens last 1 hour)
  cachedServiceToken = data.access_token;
  tokenExpiry = Date.now() + (55 * 60 * 1000);

  return cachedServiceToken;
}

// Better: Generate user-specific token
async function generateUserAPIToken(user) {
  // If Riskuity supports user impersonation
  const response = await fetch('https://api.riskuity.com/users/impersonate', {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      'X-Service-Account': process.env.RISKUITY_SERVICE_TOKEN
    },
    body: JSON.stringify({
      user_id: user.id,
      scope: 'read:projects'
    })
  });

  const data = await response.json();
  return data.access_token;
}
```

**Security:**
- ✅ Service credentials stored in Riskuity backend env vars (not Lambda)
- ✅ Token cached for 55 minutes (reduces auth calls)
- ✅ CORTAP Lambda never has credentials
- ✅ Token passed per-request
- ✅ Can implement user-specific tokens for audit trail

### Option 2: AWS Secrets Manager (If We Must Store Credentials) ⚠️

**Only if Riskuity can't implement backend token generation**

#### SAM Template Changes

```yaml
# template.yaml

Parameters:
  RiskuityServiceUsername:
    Type: String
    Default: /cortap/riskuity/service/username
    Description: SSM Parameter path for service username

  RiskuityServicePassword:
    Type: String
    Default: /cortap/riskuity/service/password
    NoEcho: true
    Description: SSM Parameter path for service password

Resources:
  # Secret for Riskuity credentials
  RiskuityCredentials:
    Type: AWS::SecretsManager::Secret
    Properties:
      Name: !Sub cortap-riskuity-creds-${Environment}
      Description: Riskuity service account credentials
      SecretString: !Sub |
        {
          "username": "${RiskuityServiceUsername}",
          "password": "${RiskuityServicePassword}"
        }

  # Lambda needs permission to read secrets
  GenerateReportFunctionRole:
    Type: AWS::IAM::Role
    Properties:
      Policies:
        - PolicyName: SecretsManagerAccess
          PolicyDocument:
            Statement:
              - Effect: Allow
                Action:
                  - secretsmanager:GetSecretValue
                Resource: !Ref RiskuityCredentials

  # Lambda function
  GenerateReportSyncFunction:
    Type: AWS::Serverless::Function
    Properties:
      Environment:
        Variables:
          RISKUITY_CREDENTIALS_SECRET: !Ref RiskuityCredentials
```

#### Lambda Code Changes

```python
# app/services/riskuity_auth.py (NEW FILE)

import boto3
import json
from datetime import datetime, timedelta
from typing import Dict, Optional

class RiskuityAuthService:
    """Manages Riskuity authentication with credential caching."""

    def __init__(self, secret_name: str):
        self.secret_name = secret_name
        self.secrets_client = boto3.client('secretsmanager')
        self._cached_token: Optional[str] = None
        self._token_expiry: Optional[datetime] = None

    async def get_api_token(self) -> str:
        """Get Riskuity API token (cached)."""

        # Return cached token if valid
        if self._cached_token and self._token_expiry > datetime.utcnow():
            return self._cached_token

        # Get credentials from Secrets Manager
        response = self.secrets_client.get_secret_value(
            SecretId=self.secret_name
        )
        credentials = json.loads(response['SecretString'])

        # Authenticate with Riskuity
        async with httpx.AsyncClient() as client:
            auth_response = await client.post(
                'https://api.riskuity.com/users/get-token',
                json={
                    'username': credentials['username'],
                    'password': credentials['password']
                }
            )
            auth_data = auth_response.json()

        # Cache token for 55 minutes
        self._cached_token = auth_data['access_token']
        self._token_expiry = datetime.utcnow() + timedelta(minutes=55)

        return self._cached_token
```

**Pros:**
- ✅ Credentials encrypted at rest
- ✅ Automatic rotation with Secrets Manager
- ✅ Access audited via CloudTrail
- ✅ Can work without Riskuity backend changes

**Cons:**
- ⚠️ Lambda still has credentials (in Secrets Manager)
- ⚠️ More complex to manage
- ⚠️ Costs ($0.40/month per secret)
- ⚠️ Still have audit trail issues (all requests as service account)
- ⚠️ Requires credential rotation process

### Option 3: Hybrid Approach (Best of Both) ✅

**Accept both service account AND user tokens**

```python
# app/api/v1/endpoints/generate.py

async def generate_report_sync(
    authorization: str = Header(None),
    request: GenerateReportRequest
):
    """
    Generate report with flexible auth:
    - If user provides token → use it (preferred)
    - If no token → use service account (fallback)
    """

    if authorization and authorization.startswith("Bearer "):
        # User-provided token (pass-through)
        token = authorization.replace("Bearer ", "").strip()
        logger.info("Using user-provided token")
    else:
        # Fallback to service account
        auth_service = RiskuityAuthService(
            secret_name=settings.riskuity_credentials_secret
        )
        token = await auth_service.get_api_token()
        logger.info("Using service account token (fallback)")

    # Rest of code is the same
    riskuity_client = RiskuityClient(
        base_url=settings.riskuity_base_url,
        api_key=token
    )
```

**Benefits:**
- ✅ Works with Riskuity backend integration (preferred)
- ✅ Falls back to service account if needed
- ✅ Gradual migration path
- ✅ Better audit trail when possible

## Recommended Approach for Production

### Phase 1: Current (Testing) ✅

**What:** Riskuity provides service account token manually
**Use:** For testing and validation
**Security:** Good enough for testing
**Implementation:** No changes needed

### Phase 2: Backend Integration (Production) ✅✅✅

**What:** Riskuity implements backend endpoint
**Flow:**
1. Frontend → Riskuity Backend → CORTAP Lambda
2. Backend generates token (service account OR user-specific)
3. Lambda receives token, passes to Riskuity API
4. No credentials in Lambda!

**Security:** Excellent
- ✅ Credentials only in Riskuity backend
- ✅ Token per request
- ✅ Proper audit trail possible
- ✅ No credential rotation in Lambda
